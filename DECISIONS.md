# Ключевые решения этапов 1–2

1. **Laravel + Blade без SPA на текущем этапе**
   - Сохранён простой серверный подход для быстрого и прозрачного развития по шагам тестового задания.

2. **Доменная модель заявки — `ServiceRequest`**
   - Используется имя `ServiceRequest`, чтобы не конфликтовать с `Illuminate\Http\Request`.

3. **Панель диспетчера выделена в отдельный контроллер и сервис**
   - Для этапа 2 добавлен отдельный `DispatcherController` и небольшой `DispatcherRequestService`.
   - Это позволяет не размазывать бизнес-логику по Blade и упростить следующий этап (workflow мастера).

4. **Правила assign/cancel заданы явно в приложении**
   - Назначение мастера (`assign`) разрешено только для статуса `new`.
   - Отмена (`cancel`) разрешена только для статусов `new` и `assigned`.
   - Недопустимые операции не выполняются и возвращают понятные сообщения об ошибке.

5. **Явная логика допустимых переходов статусов**
   - Ограничения переходов на этапе диспетчера заданы в отдельном сервисе, а не в представлениях.
   - Такое решение делает поведение предсказуемым, тестируемым и готовым к следующему шагу с конкурентными сценариями.

6. **Фильтрация заявок по enum-статусам**
   - Для фильтра на `/dispatcher` используются значения из `RequestStatus::values()`.
   - При невалидном `status` фильтр безопасно игнорируется (показываются все заявки), без падения страницы.

7. **Следующий этап отложен сознательно**
   - Master workflow (`in_progress`, `done`) и защита race condition намеренно не реализованы в этом этапе,
     чтобы сохранить поэтапную реализацию и не смешивать ответственность.

> Примечание: файл фиксирует принятые архитектурные и продуктовые решения по фактическому коду этапов 1–2.
