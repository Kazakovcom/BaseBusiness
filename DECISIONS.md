# DECISIONS

1. **Интерфейс реализован как серверное Laravel-приложение на Blade**
   - Проект не использует SPA и JS-фреймворки.
   - Все ключевые пользовательские сценарии обслуживаются обычными HTML-формами и серверными redirect/flash message.

2. **Основная доменная сущность проекта — `ServiceRequest`**
   - Для заявки используется модель `App\Models\ServiceRequest`.
   - Имя выбрано отдельно от `Illuminate\Http\Request`, чтобы не смешивать HTTP-слой и доменную модель.

3. **SQLite используется как рабочая база в локальной среде**
   - Проект хранит данные в `database/database.sqlite`.
   - Docker entrypoint подготавливает файл базы и затем запускает `migrate --seed`.

4. **Упрощённая авторизация реализована через сессию и выбор пользователя**
   - Логин выполняется на `/login` выбором пользователя из сидов.
   - В сессии хранится `auth_user_id`.
   - Доступ к ролям ограничен middleware `auth.session` и `role`.

5. **Ролевая логика диспетчера и мастера вынесена в отдельные контроллеры и сервисы**
   - Диспетчерский сценарий обслуживают `DispatcherController` и `DispatcherRequestService`.
   - Мастерский сценарий обслуживают `MasterController` и `MasterRequestWorkflowService`.
   - Blade-шаблоны остаются простыми и не содержат бизнес-логики переходов статусов.

6. **Переходы статусов жёстко ограничены по роли и текущему состоянию**
   - Диспетчер: `new -> assigned`, `new|assigned -> canceled`.
   - Мастер: `assigned -> in_progress`, `in_progress -> done`.
   - Недопустимые действия не меняют данные и возвращают контролируемое сообщение об ошибке.

7. **Мастер работает только со своими заявками**
   - На странице `/master` загружаются только записи, где `assigned_to` совпадает с текущим мастером.
   - Те же ограничения повторно проверяются в сервисе перед изменением статуса.

8. **Safe take реализован через атомарный conditional update**
   - Для операции "Взять в работу" используется единичный `update ... where id = ? and assigned_to = ? and status = assigned`.
   - Такой подход закрывает race condition: при параллельных запросах только один переводит заявку в `in_progress`, второй получает контролируемый отказ.

9. **Покрытие сосредоточено на feature- и service-уровне**
   - Тесты покрывают baseline-маршруты, создание заявки, workflow диспетчера, workflow мастера и повторный `take`.
   - Отдельно проверяются негативные сценарии и ограничения по ролям.
