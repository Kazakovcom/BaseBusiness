# Ключевые решения этапов 1–3

1. **Laravel + Blade без SPA на текущих этапах**
   - Сохранён простой серверный подход для прозрачной и поэтапной реализации тестового задания.

2. **Доменная модель заявки — `ServiceRequest`**
   - Используется имя `ServiceRequest`, чтобы не конфликтовать с `Illuminate\Http\Request`.

3. **Бизнес-логика ролей вынесена из Blade**
   - Для диспетчера используется отдельный `DispatcherController` и `DispatcherRequestService`.
   - Для мастера добавлены отдельный `MasterController` и `MasterRequestWorkflowService`.
   - Это удерживает представления простыми, а переходы статусов делает тестируемыми.

4. **Enum values статусов и ролей остаются в коде/БД, UI показывает человекочитаемые подписи**
   - В БД хранятся значения `new`, `assigned`, `in_progress`, `done`, `canceled`.
   - В интерфейсе отображаются русские подписи статусов и ролей.

5. **Переходы статусов заданы строго и по ролям**
   - Диспетчер:
     - `new -> assigned`
     - `new|assigned -> canceled`
   - Мастер:
     - `assigned -> in_progress`
     - `in_progress -> done`
   - Недопустимые переходы не выполняются и возвращают понятный контролируемый отказ.

6. **Мастер работает только со своими заявками**
   - На `/master` загружаются только записи, где `assigned_to = current master`.
   - Мастерские действия проверяют ту же принадлежность на уровне бизнес-логики, а не только интерфейса.

7. **Safe take реализован через атомарный conditional update**
   - Для операции «Взять в работу» выбран подход с атомарным `update ... where id = ? and assigned_to = ? and status = assigned`.
   - Такой вариант закрывает race condition без псевдопроверки через предварительный `if` в PHP.
   - Только один конкурентный запрос может успешно перевести заявку в `in_progress`.

8. **Complete реализован тем же строгим шаблоном проверки состояния**
   - Завершение заявки использует условное обновление с проверкой `assigned_to = current master` и `status = in_progress`.
   - Это сохраняет единый стиль переходов и исключает обход чужих заявок.

9. **Мастерские маршруты ограничены middleware по роли**
   - Маршруты `/master`, `/master/requests/{serviceRequest}/take`, `/master/requests/{serviceRequest}/complete` доступны только после `auth.session` и `role:master`.
   - Для прямых обращений предусмотрены контролируемые редиректы или явные ответы, а не silent failure.

10. **Ручная проверка race condition и финальная сдача отложены на следующий этап**
   - Финальный этап ещё включает доводку README, сценария ручной проверки гонки, возможного `race_test.sh` и итоговой упаковки репозитория.
